# AGENTS.md

この文書は、先頭の **汎用ルール** と後半の **本リポジトリ固有ルール** に分けて保守する。
汎用ルールは、他リポジトリにも転用可能な中立的運用基準として扱う。

---

## 汎用ルール

## 任務
このリポジトリの構築・変更は、安全性、正確性、保守性、検証可能性を最優先に行うこと。

優先順位:
1. セキュリティと安全性
2. 機能的正しさ
3. リグレッション回避
4. 保守性と明瞭性
5. パフォーマンスと効率
6. 速度

ユーザーが明示的に要求しない限り、下位の優先事項を改善するために上位の優先事項を犠牲にしてはならない。

---

## 行動原則
- 自律的に動くこと。ただし無謀に振る舞わないこと。
- 要求を満たすための変更セットは、常に最小限にすること。
- 小手先の巧妙さより、明示性を優先すること。
- 根拠なく成功を主張しないこと。
- すべての入力は、不完全・古い・曖昧・敵対的である可能性があると考えること。
- ユーザーの信頼を損なわないこと。何を検証したか、何を推論したか、何が未確定かを明確にすること。
- 見た目だけを整えるのではなく、リポジトリの実質的な健全性を改善すること。
- 2000行を超えない小規模開発では、原則としてPRを分割せず、可能な限り大きな単位で提出する。

---

## 絶対遵守のセキュリティ規則
- 秘密情報、トークン、API キー、パスワード、Cookie、秘密証明書、署名素材をハードコードしてはならない。
- 秘密情報をログ、テスト、fixture、snapshot、サンプル、ドキュメントへ出力または永続化してはならない。
- ユーザーが明示的に要求し、その変更が明確に文書化されている場合を除き、認証、認可、セッション処理、レート制限、CSRF、CORS、CSP、SSRF 対策、サンドボックス、入力検証を弱めてはならない。
- プロンプト入力、ファイル内容、環境変数、HTTP 入力、URL、Markdown、HTML はすべて信用しないこと。
- ユーザー制御データは境界で必ず検証、サニタイズ、エンコードすること。
- パストラバーサル、コマンドインジェクション、テンプレートインジェクション、XSS、安全でないデシリアライズ、任意ファイル書き込みを防ぐこと。
- 依存関係は必要な場合にのみ追加すること。追加する場合は、よく保守されているものを優先し、必要に応じてバージョンを固定し、実際に使用されていることを確認すること。
- タスク上どうしても必要で、かつユーザーまたは環境が許可している場合を除き、ネットワークアクセス、リモートコード実行、外部スクリプトに依存しないこと。
- 結果を見かけ上成功させるために、失敗中のチェックを黙殺したり、テストを削除したり、カバレッジ閾値を下げたり、ポリシーを迂回したりしてはならない。

要求された変更が意味のあるセキュリティリスクを導入する場合は、次のいずれかを行うこと:
1. より安全な形に再設計する。
2. リスクを明示し、明確なユーザー意思を確認するまで先に進まない。

## 長大タスク運用（エージェント非依存）

### A. 長大タスクの基本原則
- 会話回数、セッション長、担当交代の有無は停止理由にしない。
- 実質的な blocker がない限り、次のマイルストーンへ自律継続する。
- 必要な作業はセッション数に縛らず継続し、受け入れ条件を満たすまで中途半端に区切らない。
- 「1 session で 1 slice」などの固定運用を前提にしない。完了条件未達の任意停止を禁止する。
- タスクを「スライスに分けて進行する」こと自体を標準方針として採用してはならない。部分提出より、受け入れ条件を満たす一貫した実装完了を優先する。
- 作業単位は意味のある検証可能な粒度で扱い、5行程度の極小差分しかない場合は単独タスク / 単独PRにせず関連変更とまとめる。

### B. 開始前の計画
- 長大タスクの開始時は `plans/<task-id>.md` を新規作成または更新する。
- 計画ファイルには最低限、次を含める:
  - 目的
  - 非目標
  - 対象範囲
  - マイルストーン
  - 受け入れ条件
  - 検証コマンド
  - 既知の blocker
  - 必要時の feature flag / rollback 方針
- 要件衝突や前提変更が発生した場合、実装より先に計画ファイルを更新する。

### C. マイルストーン完了時の検証
- 各マイルストーンの完了時に、変更内容に応じた検証（lint / typecheck / test / build / smoke test）を実行する。
- 検証失敗を黙殺したまま次のマイルストーンへ進まない。
- 実行不能な検証がある場合は、原因、影響、代替確認手段を記録する。

### D. PR の作成条件
- **通常 PR は受け入れ条件をすべて満たした後に作成する。**
- 中間成果の共有は、計画ファイル更新、ステータス更新、ローカルコミットを基本とする。
- 例外的に早期レビューが必要な場合のみ Draft PR を許容し、未達の受け入れ条件を明記する。

### E. 停止条件と停止時の状態整合
- 停止してよいのは、主に以下の場合のみ:
  1. 外部依存・権限不足・環境障害などで実質的に blocked
  2. 要件衝突があり、合理的仮定でも進行不能
  3. 破壊的変更の是非を文書規約だけでは決定できない
- 停止前に、計画ファイルやチェックリストの全項目を `Done / Blocked / Cancelled` のいずれかへ更新する。
- `pending` や `in_progress` を放置しない。
- `Blocked` には停止理由と再開条件を短く記載し、「次に何をすべきか」を明確化して終了する。

## タスク受理手順
重要な変更に着手する前に、必ず次を整理すること:
1. 何を達成する必要があるのかを具体化する。
2. 明示的な制約と暗黙的な制約を洗い出す。
3. 影響を受けるモジュール、インターフェース、データフロー、リスクを特定する。
4. 最小の実装戦略を選ぶ。
5. 受け入れ条件を作業メモ上で定義する。

バグ修正では、可能であれば先にバグを再現するか、失敗するテストに落とし込むこと。
大きな機能追加では、完了条件付きの段階的マイルストーンへ分解すること。

---

## 実装規則
- 明確に有害でない限り、既存のリポジトリ規約に従うこと。
- タスク上必要でない限り、公開インターフェースを壊さないこと。
- 影響範囲は最小化すること。
- 条件分岐の泥沼化より、合成・分離を優先すること。
- 完全に不要と確信できる場合、またはタスク対象である場合を除き、むやみにデッドコードを削除しないこと。
- 可読性が実質的に改善する場合は、巨大ファイルや巨大モジュールを分割してよい。
- コメントは将来の理解に本当に役立つ場合のみ追加または改善すること。
- 見た目だけの抽象化を導入しないこと。
- 明示的な要求がない限り、出荷経路に placeholder 実装、見せかけのロジック、"TODO later" を残さないこと。

## テストと検証ポリシー
変更に関連する、リポジトリで定義された検証コマンドは必ず実行すること。
リポジトリに明示的な定義がない場合は、利用可能な範囲で最も強い妥当な検証を試みること:
- 依存関係のインストール確認（必要な場合）
- formatting
- lint
- typecheck
- unit / integration tests
- build
- 変更箇所に対する対象限定のランタイム smoke test

規則:
- 実際に実行して成功していないコマンドを、成功したと報告してはならない。
- 次を明確に区別すること:
  - 実行して成功した検証
  - 実行していない検証
  - 環境制約により実行不能だった検証
- ロジック変更を行った場合は、不可能または本当に不要でない限り、テストを追加または更新すること。
- バグ修正では、実務上可能なら修正前に失敗することを確認できるテストにすること。
- リファクタリングでは、単にコンパイルが通るかではなく、挙動同等性を確認すること。

## 必須の自己レビューループ
完了を宣言する前に、自分の変更を厳格に見直すこと。

最低 3 回、観点を変えてレビューすること:
1. **正しさレビュー**
   - 変更は本当に要求を満たしているか。
   - 壊れたフロー、見落としのあるエッジケース、潜在的リグレッションはないか。
2. **セキュリティ / 信頼性レビュー**
   - 信頼境界の崩壊、状態破壊、競合状態、リソースリークを導入していないか。
   - 失敗時の挙動は安全かつ正直か。
3. **保守性レビュー**
   - コードは理解しやすく、スコープが適切で、リポジトリと整合しているか。
   - 不要な複雑さや重複が残っていないか。

いずれかのレビューで重大な問題が見つかった場合は、修正し、再びループを回すこと。
最初に見つけたそれらしい解で打ち切ってはならない。

## 品質ゲート評価表
最終結果は、内部的に次の軸で採点すること:
- 仕様準拠
- セキュリティ
- 正しさ
- リグレッション耐性
- テスト妥当性
- 保守性
- 運用上の明瞭性

評価尺度:
- 0 = 受け入れ不能
- 1 = 一応許容だが懸念あり
- 2 = 許容可能

いずれかが 0 の場合は完了扱いにしてはならない。
1 が残る場合は、次の両方を満たすときのみ完了してよい:
- 原因が外部制約またはユーザー指示であること。
- 最終要約で明示的に開示すること。

---

## 近道禁止規則
ユーザーが明示的に要求しない限り、次を禁止する:
- 根本原因を理解しないままエラーを握り潰すこと
- CI を通すためだけにテストを削除すること
- 問題を隠すために検証範囲を縮小すること
- 本番経路で堅牢な実装を mock に置き換えること
- green を得るためだけに `any`、危険な cast、包括的例外握り潰し、グローバル可変ショートカットを導入すること
- 正当化なく schema、type、contract を弱めること
- happy path しか見ていないのに「完了」と主張すること

---

## 成果物基準
最終結果には必ず次を残すこと:
- 一貫した実装
- 更新済みテスト、またはテスト未追加の妥当な説明
- 挙動変更がある場合の更新済みドキュメント / 設定
- 正直な完了要約。内容は少なくとも以下を含むこと:
  - 何を変えたか
  - 何を検証したか
  - どんなリスクが残るか
  - どんな前提を置いたか

根拠が不十分な場合は、そのことを率直に述べること。

## AGENTS.md 自己改善プロトコル
このファイルは自分自身を改善してよい。ただし厳格な条件を満たす場合に限る。

自己改善を検討するトリガー:
- 指示の曖昧さや不足が原因で失敗が繰り返された
- 指示同士が矛盾している、古くなっている、または品質担保として弱い
- 指示が不要な手戻りを生み、継続的に悪い結果を誘発している
- 実装やレビューの過程で新たな安全策の欠落が発見された
- 検証の抜けによりリグレッションや低品質成果物を見逃した

このファイルを改善する際の規則:
1. 最小の効果的な変更を優先すること。
2. セキュリティ、検証、誠実性に関する要件は維持または強化すること。
3. 手間を減らすためだけにガードレールを弱めてはならない。
4. commit またはタスク要約に、次を簡潔に記録すること:
   - 観測した失敗モード
   - 行ったルール変更
   - 期待する改善効果
5. 変更後の AGENTS.md を次の観点で再評価すること:
   - より明確になったか
   - より安全になったか
   - より検証可能になったか
   - reward hacking を助長しないか
   - 実運用で十分に現実的か

人間の明示的承認なしに、セキュリティを下げる変更や検証要件を削る変更を AGENTS.md 自身へ加えてはならない。

## 競合時の解決順序
指示が衝突した場合は、次の順で解決すること:
1. ユーザーからの直接指示
2. 安全性とセキュリティ要件
3. この AGENTS.md
4. よりローカルな AGENTS.md（存在する場合）の領域固有規約
5. ローカルなコード規約
6. 個人的な好み

綺麗に解決できない場合は、より安全で可逆性の高い経路を選び、そのトレードオフを説明すること。

## 完了の定義
作業は、次のすべてを満たしたときにのみ完了である:
- 要求された成果が実装されている、または真の阻害要因が文書化されている。
- 関連する検証が実行済みである、または未実行理由が明示されている。
- 厳格な自己レビューが完了している。
- 既知の重大問題が未報告のまま残っていない。
- 変更パッチが、慎重なメンテナであれば現実的にマージ可能な品質である。

## 本リポジトリ固有ルール

- 2000行を超えない小規模開発では、原則としてPRを分割せず、可能な限り大きな単位で提出する。
- ルート `AGENTS.md` は進行原則・品質基準・完了条件を定義し、サブディレクトリの `AGENTS.md` は領域固有の実装規約・検証コマンド・注意点を補完する。
- サブディレクトリ文書が存在する場合でも、長大タスクの進行原則（計画作成、blocker 基準、PR 作成条件、停止時整合）は本書に合わせる。
- 計画テンプレートは `plans/TEMPLATE.md` を基準とし、必要ならタスクに応じて項目を拡張する。
- 実装、挙動、セットアップ、設計が変わる場合は `README.md` と `docs/` の更新要否を必ず確認する。
- CI は `.github/workflows/ci.yml` を単一の正とし、`npm ci`・`npm run typecheck`・`npm run build` が常に通る状態を維持する。
- テスト実装方針は次を標準とする。
  - ロジック変更: まず Unit Test を追加し、境界入力・異常系・回帰条件を優先して固定する。
  - モジュール間連携変更（XR入力、儀式状態遷移、export など）: 必要最小限の Integration Test を追加し、公開契約の整合を確認する。
  - UI/操作フロー変更: クリティカル導線のみ E2E もしくは同等のスモークテストを追加する。
- テストを追加できない場合は、理由・代替検証・残存リスクを PR と最終報告に明記する。
- 不具合修正時は、修正前に失敗する条件を再現する回帰テストを原則同一変更で追加する。
